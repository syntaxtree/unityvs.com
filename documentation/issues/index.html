Known issues

  • When being debugged, Unity's scripting engine is known to crash occasionally. Unity 4 should be more stable in this regard.
  • Sometimes when debugging Unity running on OS X, when stepping, Unity hangs. We're working with Unity to get this fixed.
  • Unity can freeze when recompiling scripts while being debugged. It's not really a UnityVS bug, but it can lead to Visual Studio hanging until Unity is closed.
  • When clicking in the Unity Console, UnityVS only open the file that contains the error or warning, but UnityVS doesn't take you to the exact line. This is because Unity currently only gives us the file name, and no line number information. Until we can get the line number as well, there's an and unsupported way to enable them by using ufixit. Alternatively, you can disable the use of our file opener in Unity by using the UnityVS -> Configuration menu, switch back to MonoDevelop as an external script editor and use the Unity External Script Editor Helper, which redirects calls to MonoDevelop by pointing it to our file opener with a configuration as follows: 
    [Settings]
Application=C:\Program Files (x86)\UnityVS\2012\UnityVS.OpenFile.exe
Arguments="*file*" *linenumber*

    If you can use Unity 4.2, there's a much easier solution, see Script opening with Unity 4.2.

  • Stepping in and out of methods is sometimes capricious. This is related to a couple of bugs in Unity's scripting engine.
  • Resharper blocks our auto-indentation for .js files.

For all issues that can not be addressed or worked around from UnityVS, we're collaborating upstream to ensure they'll fixed promptly.

Please do report all crashes and bugs to the mailing list, the discussion room or by email. Thanks!

What to do when Visual Studio locks up?

We've had reports that Visual Studio sometimes locks up when debugging. We find it unacceptable. We also need data to help fix those. Here's what you can do to help us whenever Visual Studio is locked up.

  • Open another Visual Studio instance,
  • In the new Visual Studio instance, do Tools -> Attach to Process,
  • Select and attach the debugger to the Visual Studio instance that is locked up,
  • Pause the debugger,
  • Go to the Commands window pane, if you don't have go to View -> Other Windows -> Command Window,
  • In the Command window pane input the command: 
    Debug.ListCallStack /AllThreads /ShowExternalCode
  • Send us the whole thread dump, along with a description of when the lock up at unityvs@syntaxtree.com,

This will help us investigate and fix those issues.
